# Library Design Constraints for LLM

本文档用于约束 LLM 设计 TypeScript/JavaScript 库、SDK、框架、工具包、插件系统、内部基础设施包时的设计判断。

重点不是业务流程，而是：

- public API 是否稳定
- 参数和配置结构是否清晰
- 类型契约是否可靠
- 扩展点是否受控
- 内部实现是否可替换
- 调用方是否不容易误用
- 后续版本是否容易演进

---

## 总原则

```ts
/**
 * Library Design Principles
 *
 * - Public API is a product.
 *   公共 API 是库的产品，不是内部实现的泄漏。
 *
 * - Make invalid usage hard.
 *   好的库设计应该让错误用法变难，而不是只靠文档提醒。
 *
 * - Optimize for call-site readability.
 *   调用点应该尽量自解释。
 *
 * - Keep the public surface small.
 *   暴露越多，兼容性成本越高。
 *
 * - Design for extension, not mutation of internals.
 *   调用方应该通过受控扩展点定制行为，而不是依赖内部对象结构。
 */
```

## Public API 必须小而稳定

```ts
/**
 * 约束
 *
 * - 不要随手 export 内部函数、内部类型、内部状态。
 * - public API 只暴露调用方真正需要依赖的能力。
 * - internal/helper/adapter/debug API 默认不暴露。
 *
 * 理由
 *
 * 对库来说，任何 export 都会变成兼容性承诺。
 * 暴露越多，未来重构越困难。
 */

// bad
export {
  createEngine,
  normalizeOptions,
  createInternalCacheKey,
  parseUnsafeNode,
  applyLowLevelPatch,
  EngineRuntimeState,
};

// good
export {
  createEngine,
  defineEngineConfig,
  definePlugin,
  type Engine,
  type EngineConfig,
  type EnginePlugin,
};
```

```ts
/**
 * 目录层面也要区分 public / internal。
 */

// good
src / index.ts; // only stable public exports
public / createEngine.ts;
defineEngineConfig.ts;
definePlugin.ts;
types.ts;
core / engine.ts;
scheduler.ts;
internal / resolveEngineConfig.ts;
createCacheKey.ts;
debug.ts;
adapters / node.ts;
browser.ts;
```

## 参数与 options 必须表达语义结构

```ts
/**
 * 约束
 *
 * - 超过 3 个参数，优先对象化。
 * - 多个同类型参数，必须对象化。
 * - 裸 boolean 参数禁止。
 * - 对象参数不是万能 options，不同语义域必须继续分组。
 * - 核心输入、执行策略、运行时依赖、诊断能力应分层。
 *
 * 理由
 *
 * 位置参数依赖记忆，对调用方和 LLM 都不友好。
 * 平铺 options 会让字段来源混乱、默认值不清、扩展困难。
 */

// bad
createEngine(source, "esm", "cjs", true, false, 3000);

// bad
createEngine({
  source,
  parser,
  logger,
  debug,
  signal,
  timeoutMs,
  retries,
  format,
  cwd,
  env,
});

// good
createEngine({
  input: {
    source,
    format: "esm",
  },
  output: {
    format: "cjs",
  },
  execution: {
    cwd,
    env,
    signal,
    timeoutMs: 3000,
    retries: 3,
  },
  diagnostics: {
    logger,
    level: "warn",
  },
});
```

```ts
/**
 * boolean 参数优先替换为语义枚举。
 */

// bad
parse(input, true);

// acceptable
parse(input, {
  strict: true,
});

// better
parse(input, {
  mode: "strict",
});

type ParseMode = "strict" | "loose" | "recover";
```

## 默认值、defineConfig 与配置优先级必须明确

```ts
/**
 * 约束
 *
 * - 可选字段必须有明确默认语义。
 * - public config/options 的可选字段必须用 JSDoc `@default` 标明默认值或默认语义。
 *   如果项目统一使用 TSDoc / TypeDoc，则可统一改用 `@defaultValue`。
 * - public 配置入口优先提供 `defineXXXConfig`。
 * - 默认值应集中定义，不要散落在调用路径上的 `??` / `||` 中。
 * - 不要让 undefined 同时表示多个含义。
 *
 * 理由
 *
 * 库的默认行为会影响所有调用方。
 * defineConfig 让调用点、类型推导、默认语义和文档更容易对齐。
 */

// bad
type EngineOptions = {
  cache?: boolean;
  debug?: boolean;
  timeout?: number;
};

// good
type EngineConfig = {
  /**
   * Cache behavior used when the caller does not override it.
   *
   * @default "inherit"
   */
  cache?: {
    /**
     * @default "inherit"
     */
    mode?: "inherit" | "disabled" | "read-only" | "read-through";
  };

  /**
   * Diagnostic output behavior.
   *
   * @default { level: "error" }
   */
  diagnostics?: {
    /**
     * @default "error"
     */
    level?: "silent" | "error" | "warn" | "debug";
  };

  /**
   * Execution limits.
   *
   * @default { timeoutMs: 10_000 }
   */
  execution?: {
    /**
     * Maximum time allowed for one execution.
     *
     * @default 10_000
     */
    timeoutMs?: number;
  };
};

export function defineEngineConfig(config: EngineConfig): EngineConfig {
  return config;
}

const builtInEngineConfig = defineEngineConfig({
  cache: {
    mode: "inherit",
  },
  diagnostics: {
    level: "error",
  },
  execution: {
    timeoutMs: 10_000,
  },
});
```

```ts
/**
 * 配置优先级必须固定，并写进 public 文档。
 *
 * 推荐顺序：
 * built-in default < project config < call-site override < runtime override
 */

const projectConfig = defineEngineConfig({
  diagnostics: {
    level: "warn",
  },
});

createEngine({
  config: projectConfig,
  execution: {
    timeoutMs: 30_000,
  },
});
```

## 类型契约与返回值必须让错误用法难以表达

```ts
/**
 * 约束
 *
 * - 不要用一堆 optional 字段表达互斥状态。
 * - 用 discriminated union 表达模式、状态、结果。
 * - 能用 literal union 就不要用裸 string。
 * - 除非是坐标、范围、底层协议等天然 tuple，否则复杂返回值必须对象化。
 *
 * 理由
 *
 * 库 API 的类型就是文档和约束。
 * tuple 依赖顺序，不利于阅读、扩展和版本演进。
 */

// bad
type BuildResult = {
  success: boolean;
  output?: string;
  error?: Error;
};

// good
type BuildResult =
  | {
      ok: true;
      output: string;
      warnings: BuildWarning[];
      meta: {
        durationMs: number;
        cacheHit: boolean;
      };
    }
  | {
      ok: false;
      error: BuildError;
      warnings: BuildWarning[];
      meta: {
        durationMs: number;
      };
    };
```

```ts
// bad
const [output, warnings, meta] = await compile(input);

// good
const result = await compile(input);

if (result.ok) {
  result.output;
  result.warnings;
  result.meta;
}
```

## 错误必须稳定、可分支处理

```ts
/**
 * 约束
 *
 * - 不要只 throw Error(message)。
 * - 错误应有稳定 code。
 * - message 面向人，code 面向程序，details 面向调试。
 *
 * 理由
 *
 * 调用方不能依赖 message 做逻辑判断。
 * message 会变，code 才能稳定兼容。
 */

// bad
throw new Error("invalid config");

// good
throw new LibraryError({
  code: "INVALID_CONFIG",
  message: "Invalid configuration.",
  details: {
    field: "output.format",
    expected: ["esm", "cjs"],
    received: "unknown",
  },
});
```

```ts
type LibraryErrorCode =
  | "INVALID_CONFIG"
  | "PARSE_FAILED"
  | "PLUGIN_FAILED"
  | "TIMEOUT"
  | "UNSUPPORTED_RUNTIME";

class LibraryError extends Error {
  readonly code: LibraryErrorCode;
  readonly details?: unknown;
}
```

## 扩展点与插件能力必须受控

```ts
/**
 * 约束
 *
 * - 不要在 options 中散落大量 onXXX callback。
 * - 扩展点应收敛为 plugin / middleware / adapter。
 * - 生命周期阶段必须明确。
 * - 插件不应直接修改 engine 内部私有状态。
 * - 插件能力应通过 ctx 暴露的受控 API 完成。
 *
 * 理由
 *
 * callback sprawl 会导致顺序不清、组合困难、调试困难。
 * 插件系统一旦允许任意访问内部对象，就等于把内部实现变成 public API。
 */

// bad
createEngine({
  onStart,
  onResolve,
  onTransform,
  onCacheHit,
  onCacheMiss,
  onError,
  onFinish,
});

// good
createEngine({
  plugins: [cachePlugin(), diagnosticsPlugin(), transformPlugin()],
});
```

```ts
interface EnginePlugin {
  name: string;

  setup?(ctx: PluginSetupContext): void | Promise<void>;

  beforeRun?(ctx: RunContext): void | Promise<void>;

  transform?(
    input: TransformInput,
    ctx: TransformContext,
  ): TransformOutput | Promise<TransformOutput>;

  onError?(error: unknown, ctx: ErrorContext): void | Promise<void>;

  finally?(ctx: RunContext): void | Promise<void>;
}

interface PluginSetupContext {
  hooks: HookRegistry;
  cache: CacheController;
  diagnostics: Diagnostics;
  runtime: {
    platform: "node" | "browser" | "worker";
  };
}
```

## Core 与 Adapter 必须分离

```ts
/**
 * 约束
 *
 * - core 不应硬编码 Node、浏览器、文件系统、HTTP、数据库。
 * - runtime-specific 能力放 adapter。
 * - core 尽量保持纯粹、可测试、可复用。
 *
 * 理由
 *
 * 库经常需要支持多运行时。
 * 核心逻辑依赖具体环境会降低可移植性。
 */

// bad
async function compile(input: string) {
  const config = await fs.promises.readFile("config.json", "utf8");
  return runCompiler(input, config);
}

// good
async function compile(
  input: string,
  options: {
    configLoader: ConfigLoader;
  },
) {
  const config = await options.configLoader.load();
  return runCompiler(input, config);
}
```

```ts
interface ConfigLoader {
  load(): Promise<Config>;
}

class NodeFileConfigLoader implements ConfigLoader {
  async load() {
    // Node fs implementation
  }
}

class MemoryConfigLoader implements ConfigLoader {
  async load() {
    // test/browser implementation
  }
}
```

## 内部结构和字符串协议不要外泄

```ts
/**
 * 约束
 *
 * - cache key、dedupe key、task key、plugin internal id 应由库内部生成。
 * - 调用方提供语义输入，不提供内部协议。
 * - 除非是成熟标准格式，否则不要用字符串承载复杂结构。
 *
 * 理由
 *
 * 内部 key 一旦外泄，调用方会依赖它。
 * 字符串协议难校验、难补全、难重构、难被 LLM 正确生成。
 */

// bad
cache.get("compile:src/index.ts:esm:prod");

// good
cache.get({
  type: "compile",
  input: {
    file: "src/index.ts",
    format: "esm",
    mode: "production",
  },
});
```

```ts
// bad
transform({
  pipeline: "parse|minify|emit:cjs",
});

// good
transform({
  pipeline: [{ type: "parse" }, { type: "minify" }, { type: "emit", format: "cjs" }],
});
```

```ts
/**
 * 可以接受字符串协议的场景：
 *
 * - URL
 * - glob
 * - RegExp source
 * - cron
 * - CSS selector
 * - GraphQL SDL
 * - SQL inside adapter
 * - 已有成熟生态的 DSL
 *
 * 即便如此，也应考虑提供结构化替代。
 */
```

## 命名必须表达抽象层级和副作用

```ts
/**
 * 约束
 *
 * - public API 避免 run / handle / process / do 这类模糊命名。
 * - 名称应体现库的领域抽象。
 * - 同类操作命名保持一致。
 * - get/parse/resolve/normalize 这类函数不应产生隐藏副作用。
 * - 会写入、注册、启动、订阅、释放资源的 API，命名必须显式。
 *
 * 理由
 *
 * 库调用方非常依赖函数名判断成本和副作用。
 * 命名越模糊，文档依赖越重。
 */

// bad
run();
process();
handle();
exec();
getConfig(); // secretly reads file, writes cache, starts watcher

// good
parseInput();
transformInput();
emitOutput();

loadConfig();
resolveConfig();
startConfigWatcher();
```

```ts
/**
 * 命名应成组一致。
 */

createEngine();
defineEngineConfig();
definePlugin();
defineAdapter();

loadConfig();
resolveConfig();
normalizeConfig();
```

## 配置、实例与公共输入输出必须分清

```ts
/**
 * 约束
 *
 * - 配置对象描述行为。
 * - 实例对象承载状态和生命周期。
 * - 不要让配置对象在运行中被隐式修改。
 * - 不要修改调用方传入的对象。
 * - public API 返回的数据不要要求调用方原地修改。
 *
 * 理由
 *
 * 配置应可复用、可序列化、可比较。
 * 隐式 mutation 是库设计中最常见的惊讶来源。
 */

// bad
function normalizeConfig(config: EngineConfig) {
  config.mode ??= "production";
  return config;
}

// good
function normalizeConfig(config: EngineConfig): ResolvedEngineConfig {
  return {
    ...config,
    mode: config.mode ?? "production",
  };
}
```

```ts
// good
const engine = await createEngine({
  config: projectConfig,
});

try {
  await engine.run(input);
} finally {
  await engine.dispose();
}
```

## Async API 与资源生命周期必须显式

```ts
/**
 * 约束
 *
 * - 长耗时 async API 应支持 AbortSignal。
 * - 持有资源的对象必须提供 dispose/close/stop。
 *
 * 理由
 *
 * 库不能假设调用方永远会等任务自然完成。
 * 不支持取消会导致泄漏、卡死、测试困难。
 */

await compile({
  input,
  execution: {
    signal: abortController.signal,
    timeoutMs: 30_000,
  },
});
```

```ts
const watcher = createWatcher({
  paths: ["src"],
});

watcher.on("change", callback);

await watcher.dispose();
```

## 性能边界必须显式

```ts
/**
 * 约束
 *
 * - 不要为了微优化牺牲 API 清晰度。
 * - 重复计算、重复 IO、无界缓存、同步阻塞都要显式控制。
 * - cache、batch、concurrency、queue、pool、ttl、maxSize 等边界必须可见。
 *
 * 理由
 *
 * 库会被放进未知规模的调用场景。
 * 结构性低效会被调用方放大。
 */

// bad
const cache = new Map<string, unknown>();

// good
const cache = createCache({
  maxSize: 1000,
  ttlMs: 60_000,
});
```

```ts
// better
function transformMany(inputs: string[]) {
  const sharedContext = createTransformContext();

  return inputs.map((input) =>
    transform(input, {
      context: sharedContext,
    }),
  );
}
```

## 文档应解释约束，而不是复述代码

```ts
/**
 * 约束
 *
 * - public API 必须有简短文档。
 * - 文档重点解释语义、默认值、副作用、错误、生命周期。
 * - 不要写“把 x 设置为 x”这种重复代码的注释。
 *
 * 理由
 *
 * 库文档是 API 契约的一部分。
 */

// bad
/**
 * Set timeoutMs.
 */
timeoutMs?: number

// good
/**
 * Maximum time allowed for one execution.
 *
 * @default 10_000
 * When exceeded, the operation fails with TIMEOUT.
 */
timeoutMs?: number
```

## LLM 生成库设计时的检查清单

```ts
/**
 * Public API
 *
 * - 是否暴露了过多内部函数？
 * - export 是否都是稳定承诺？
 * - 是否把内部 key / 内部状态 / 内部结构泄漏给调用方？
 */

/**
 * 参数与配置
 *
 * - 是否存在超过 3 个位置参数？
 * - 是否存在多个同类型位置参数？
 * - 是否存在裸 boolean？
 * - 同类语义是否已经分组？
 * - options 是否变成垃圾桶？
 * - public config/options 的可选字段是否用 JSDoc `@default` 或 TSDoc / TypeDoc `@defaultValue` 标明默认值或默认语义？
 * - 是否应该提供 `defineXXXConfig`？
 * - 配置优先级是否明确？
 */

/**
 * 类型与结果
 *
 * - 是否用 string 替代了可枚举 literal union？
 * - 是否用 optional 字段表达互斥状态？
 * - 返回值是否依赖 tuple 顺序？
 * - 错误是否有稳定 code？
 */

/**
 * 扩展与运行时
 *
 * - 是否 callback 满天飞？
 * - 是否有清晰 plugin/middleware/adapter 模型？
 * - 插件是否能访问不该访问的内部状态？
 * - 生命周期阶段是否明确？
 * - 长耗时任务是否支持 AbortSignal？
 * - 持有资源的对象是否支持 dispose/close/stop？
 * - 是否隐式修改调用方传入的对象？
 */

/**
 * 性能与文档
 *
 * - cache、batch、concurrency、queue、pool、ttl、maxSize 等边界是否可见？
 * - 文档是否解释语义、默认值、副作用、错误和生命周期？
 * - 注释是否只是复述代码？
 */
```
