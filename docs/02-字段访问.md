# 02. 字段访问

## Accessor Graph

Codegen 根据 GraphQL schema 生成 typed accessor。TypeScript schema 类型交给 GraphQL Code Generator 生成；GQLens 只生成 accessor、schema contract 和 invalidation spec。

Accessor **不是数据对象**，而是数据需求的镜头：即使数据尚未返回，读取终点也必须能声明 selection，并订阅 store 中当前可用的值。

生成规则只由 schema 决定：

- 无 args 字段 → getter
- 有 args 字段 → 函数，即使所有 args 都是可选项
- scalar 字段 → 读取终点
- entity 字段 → 返回下一层 accessor
- entity list 字段 → 返回列表 identity accessor

### Root Accessor

```ts
type QueryNode = {
  user(args: { id: string }): UserNode;
  get viewer(): UserNode;
  todo(args: { id: string }): TodoNode;
  todos(args?: TodoFilter): TodoListNode;
};
```

这个形态是 contract，不是手写约定。运行时不根据字段名猜测 accessor 类型。

调用 `q.user({ id })` 返回 `UserNode` accessor，不返回 `User` 数据对象，也不订阅整个 `User:<id>`。响应式读取只发生在后续 scalar 字段或列表 identity 上。

### Entity Accessor

```ts
type UserNode = {
  get id(): string;
  get name(): string;
  get avatar(): string | null;
  get online(): boolean;
  posts(args?: PostArgs): PostListNode;
};
```

### 读取契约

```ts
const q = useQuery();

q.viewer.name; // getter 链，无括号
q.viewer.avatar; // getter
q.user({ id: "1" }).name; // user 有参→()，name 无参→getter
q.user({ id: "1" }).online;
```

一次调用同时产生：

```
记录 selection：Query.viewer.name
读取 signal：  Query.viewer slot → User:1.name（若已解析）
```

首次读取时 `Query.viewer` 可能还没有解析到具体实体，因此读取链路允许中间值是 root / relation slot。响应返回后，slot 指向 `User:1`，之后所有访问都共享同一个 `User:1.name` field signal。

## 字段函数的两种形态

### 标量字段（终点）

```ts
// 无参标量 → getter
get name(): string {
  ctx.demand(path.field("name"))
  return ctx.readField({ owner: { kind: "entity", ref }, path: [{ field: "name" }] })
}
```

标量字段是读取链的终点。它返回稳定、可直接渲染的字段值；不得返回临时对象来承载响应式状态。

### 字段值状态

字段 getter 只返回字段值本身，不返回 `{ data, loading, error }` 一类包装对象。运行时必须区分四种状态：

| 状态    | 返回值 / 表现        | 含义                                     |
| ------- | -------------------- | ---------------------------------------- |
| missing | `undefined`          | store 尚无值、slot 未解析、分支不匹配    |
| null    | `null`               | 服务端明确返回 `null`                    |
| stale   | 返回旧值             | store 有值但已过期或被 invalidation 标记 |
| loading | `q.loading === true` | session 存在 in-flight request           |

`undefined` 不等价于 `null`。`null` 是 GraphQL 数据语义；`undefined` 是客户端尚不能给出该字段值。

stale 不会让字段瞬间变回 `undefined`。旧值仍可用于当前渲染，active demand 由 session 调度后台 refetch。

TypeScript 返回类型必须反映 missing：

```ts
q.viewer.name; // string | undefined
q.viewer.avatar; // string | null | undefined
```

框架适配层可以提供 Suspense / required mode，把 missing 转换为 throw promise 或边界状态；这属于读取策略，不改变 accessor 的底层值语义。

### 关联字段（推进路径）

```ts
// 有参关联 → 函数
posts(args?: PostArgs): PostListNode {
  const nextPath = path.field("posts", args)
  return createPostListNode(ctx, ref.relation("posts", args), nextPath)
}

// 无参关联 → getter
get viewer(): UserNode {
  return createUserNode(ctx, ref, path.field("viewer"))
}
```

关联字段不是“取出一个对象”，而是把读取路径向前推进一步。它返回下一个 accessor，让 UI 可以继续表达更细的字段需求。具体 demand 在下游 scalar 字段或列表 identity 被读取时形成。

```ts
q.viewer.name;
q.post({ id }).author.avatar;
```

这两个例子都不会要求订阅整个 `viewer` 或 `author` 对象；真正订阅的是最终被读取的字段，以及中间路径的 identity slot。

## 列表访问模式

列表拆分为「身份」和「字段」两个层次。普通实体列表暴露 `ids`：

```ts
// 获取 ID 列表
const ids = q.todos({ done: false }).ids; // → readonly string[] | undefined

// 按 ID 读取单个实体的字段
const title = q.todo({ id }).title; // → string | undefined
```

列表 identity 必须包含 root / relation 名称、args、排序和分页窗口。`q.todos({ done: false }).ids` 与 `q.todos({ done: true }).ids` 是两条不同的列表需求，即使其中部分实体相同，实体字段仍通过 GraphDataStore 共享。

列表字段不生成 `item(id)` / `node(id)` 之类的快捷入口。行字段必须通过显式 root accessor 重新进入。这个限制保持了列表 identity 与 entity field 的边界：列表只负责成员和顺序，实体字段只通过 entity identity 读取。

如果列表元素是 interface / union，identity 必须包含 `__typename`：

```ts
const refs = q.search({ text }).refs; // → readonly EntityRef[] | undefined
```

abstract list 不应只暴露 `ids`。不同 concrete type 可能共享同一个 `id` 值；`refs` 才能无歧义地表达成员与顺序。

abstract list 的行字段仍要重新进入 accessor graph。调用方应根据 `ref.__typename` 进入具体 root accessor，或进入 schema 已定义的 abstract root accessor 后使用 `$on`。不得为列表生成 `item(ref)` / `node(ref)` 快捷入口。

订阅粒度规则：

- 列表组件只订阅 identity 列表
- 行组件按 ID / ref 重新进入 accessor 读取字段
- 字段组件只订阅单个字段
- 不订阅整个对象
- 不返回临时对象作为响应式返回值

分页或 connection 必须继续沿用同一思想：

```ts
const page = q.todos({ done: false, first: 20 });

page.ids;
page.pageInfo.hasNextPage;
```

`ids` 表达列表成员与顺序，`pageInfo` 表达分页状态。它们都属于列表 identity，不属于某个 Todo 实体字段。

## Interface / Union 访问

interface / union 的类型分支必须由 codegen 显式生成，不能依赖运行时属性探测。

共同字段按普通字段访问：

```ts
q.viewer.pets({ first: 10 }).refs;
q.pet({ id }).name;
q.pet({ id }).__typename;
```

类型分支字段通过生成出的 inline-fragment accessor 访问：

```ts
q.pet({ id }).$on.Cat.meows;
q.pet({ id }).$on.Dog.barks;
```

`$on.<TypeCondition>` 表达 GraphQL `... on <TypeCondition>`。type condition 通常是 concrete object type，也可以是当前 abstract type 可接受的 interface condition。

设计约束：

- `$on` 不是 Proxy；可选 type condition 来自 schema possibleTypes / interface 实现关系
- 读取 `$on.Cat.meows` 记录 `... on Cat { meows }`
- Planner 必须把类型分支渲染为 inline fragment
- store 写入仍按真实 `__typename + id + field` 归一
- 当前实体 `__typename` 与分支不匹配时，分支字段读取返回 `undefined`，不得伪造默认值

这使 union / interface 与普通 entity 字段共享同一套 normalized field signal，同时保留 GraphQL 类型收窄语义。

其他 GraphQL 语义的映射见 [GraphQL 语义映射](./参考-GraphQL语义映射.md)。

## Codegen 职责

Codegen 必须生成：

| 产物                       | 用途                                             |
| -------------------------- | ------------------------------------------------ |
| Schema 类型                | GraphQL 类型在宿主语言的映射                     |
| Root accessor              | 无参字段 getter + 有参字段函数                   |
| Entity node accessor       | 标量 getter + 关联字段函数/ getter               |
| List identity accessor     | 普通列表 `ids`；abstract 列表 `refs`             |
| Args 类型                  | 参数对象的类型定义                               |
| Interface / union 分支     | `$on.<TypeCondition>` inline-fragment accessor   |
| Schema contract            | 标识实体主键，并描述响应数据到实体字段的映射规则 |
| Mutation wrapper 类型      | 带 optimistic / invalidates 的函数               |
| Mutation accessor（`api`） | `api.user.rename` 等 typed mutation 入口         |
| Invalidation 类型          | 每个 entity+field 的字面量 union                 |

其中 schema contract 使 Planner 能自动补齐 `id` 和 `__typename`，用户无需手动声明。

Codegen 还需要提供 schema contract，让 runtime 知道：

- 哪些字段返回实体，哪些字段返回列表
- 哪些字段是 scalar，哪些字段只是 relation slot
- 哪些类型的 identity 来自 `id + __typename`，哪些需要用户配置

Planner 和 store 不应靠运行时值猜测这些信息。

## AST 编译插件（可选）

编译器插件纯粹是性能 / DX 优化层，**不是正确性的前提**。它可以做：

- 预计算 selection path
- 生成稳定的 operation name
- 提取变量形状
- 生成 persisted query hash
- 开发期静态诊断非法字段访问
- tree-shaking 辅助
