# 02. 字段访问

## Accessor Graph

Codegen 根据 GraphQL schema 生成 typed accessor。它**不是数据对象**，而是一个数据需求的镜头：即使数据尚未返回，也能通过字段访问声明 selection，并在 cache 中订阅当前可用的值。

Accessor 的目标是让 UI 写出“它需要什么”，而不是手动拼 query、拆 response、管理局部 loading。

### Root Accessor

```ts
type QueryNode = {
  user(args: { id: string }): UserNode;
  get viewer(): UserNode;
  todo(args: { id: string }): TodoNode;
  todos(args?: TodoFilter): TodoListNode;
};
```

无参字段使用 getter、有参字段使用函数——以此为统一的语法规则。

### Entity Accessor

```ts
type UserNode = {
  get id(): string;
  get name(): string;
  get avatar(): string | null;
  get online(): boolean;
  posts(args?: PostArgs): PostListNode;
};
```

### 使用示例

```ts
const q = useQuery();

q.viewer.name; // getter 链，无括号
q.viewer.avatar; // getter
q.user({ id: "1" }).name; // user 有参→()，name 无参→getter
q.user({ id: "1" }).online;
```

一次调用同时产生：

```
记录 selection：Query.viewer.name
读取 signal：  Query.viewer slot → User:1.name（若已解析）
```

首次读取时 `Query.viewer` 可能还没有解析到具体实体，因此读取链路允许中间值是 root / relation slot。响应返回后，slot 指向 `User:1`，之后所有访问都共享同一个 `User:1.name` field signal。

## 字段函数的两种形态

### 标量字段（终点）

```ts
// 无参标量 → getter
get name(): string {
  ctx.select(path.field("name"))
  return ctx.cache.field(ref, "name").read()
}
```

标量字段是读取链的终点。它应该返回一个稳定、可直接渲染的值，而不是临时对象。

### 关联字段（继续链式访问）

```ts
// 有参关联 → 函数
posts(args?: PostArgs): PostListNode {
  const nextPath = path.field("posts", args)
  ctx.select(nextPath)
  return createPostListNode(ctx, ref.relation("posts", args), nextPath)
}

// 无参关联 → getter
get viewer(): UserNode {
  ctx.select(path.field("viewer"))
  return createUserNode(ctx, ref, path.field("viewer"))
}
```

关联字段不是“取出一个对象”，而是把读取路径向前推进一步。它返回下一个 accessor，让 UI 可以继续表达更细的字段需求。

```ts
q.viewer.name;
q.post({ id }).author.avatar;
```

这两个例子都不会要求订阅整个 `viewer` 或 `author` 对象；真正订阅的是最终被读取的字段，以及中间路径的 identity slot。

## 列表访问模式

列表拆分为「身份」和「字段」两个层次：

```ts
// 获取 ID 列表
const ids = q.todos({ done: false }).ids; // → string[]

// 按 ID 读取单个实体的字段
const title = q.todo({ id }).title; // → string
```

列表 identity 必须包含 root / relation 名称、args、排序和分页窗口。`q.todos({ done: false }).ids` 与 `q.todos({ done: true }).ids` 是两条不同的列表需求，即使其中部分实体相同，实体字段仍通过 normalized cache 共享。

推荐原则：

- 列表组件只订阅 ID 列表
- 行组件按 ID 读取字段
- 字段组件只订阅单个字段
- 不订阅整个对象
- 不返回临时对象作为响应式返回值

分页或 connection 可以继续沿用同一思想：

```ts
const page = q.todos({ done: false, first: 20 });

page.ids;
page.pageInfo.hasNextPage;
```

`ids` 表达列表成员与顺序，`pageInfo` 表达分页状态。它们都属于列表 identity，而不是某个 Todo 实体字段。

## Codegen 职责

Codegen 必须生成：

| 产物                       | 用途                                     |
| -------------------------- | ---------------------------------------- |
| Schema 类型                | GraphQL 类型在宿主语言的映射             |
| Root accessor              | 无参字段 getter + 有参字段函数           |
| Entity node accessor       | 标量 getter + 关联字段函数/ getter       |
| Args 类型                  | 参数对象的类型定义                       |
| Entity identity 元数据     | 标识实体主键的字段（id/\_\_typename）    |
| Normalizer 元数据          | 响应数据到实体字段的映射规则             |
| Mutation wrapper 类型      | 带 optimistic / invalidates 的函数       |
| Mutation accessor（`api`） | `api.user.rename` 等 typed mutation 入口 |
| Invalidation 类型          | 每个 entity+field 的字面量 union         |

其中 entity identity 元数据使 Planner 能自动补齐 `id` 和 `__typename`，用户无需手动声明。

Codegen 还需要提供 schema 元数据，让 runtime 知道：

- 哪些字段返回实体，哪些字段返回列表
- 哪些字段是 scalar，哪些字段只是 relation slot
- 哪些类型的 identity 来自 `id + __typename`，哪些需要用户配置

Planner 和 cache 不应靠运行时值猜测这些信息。

## AST 编译插件（可选）

编译器插件纯粹是性能 / DX 优化层，**不是正确性的前提**。它可以做：

- 预计算 selection path
- 生成稳定的 operation name
- 提取变量形状
- 生成 persisted query hash
- 开发期静态诊断非法字段访问
- tree-shaking 辅助
