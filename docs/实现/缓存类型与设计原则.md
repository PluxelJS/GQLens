# 缓存类型与设计原则

本文用于区分 GQLens 内部几类 cache / store。详细持久化和外部 records map 设计见 [GraphDataStore 持久化设计](./GraphDataStore持久化设计.md)。

## 总原则

- 业务数据使用 `GraphDataStore`。
- `GraphDataStore` 存的是 GraphQL 业务数据，只是内部采用 normalized field / slot 布局。
- 派生缓存只缓存可重算结果，丢弃后不改变 UI 业务语义。
- TTL 表示 freshness，不表示删除。
- 非大基数业务数据不做无策略淘汰；需要容量控制时通过 `GraphDataRecordMap` 的外部策略实现。

## GraphDataStore

旧 `NormalizedCache` 命名已移除，统一使用 `GraphDataStore`。

职责：

- 保存 GraphQL field / slot records。
- 维护 signal 通知。
- 处理 TTL、invalidation、transaction rollback。
- 支撑 query、live query、mutation 写入同一套业务数据。

records 分为：

- `fields`：实体字段或 embedded leaf，例如 `User:1.name`。
- `slots`：root / relation / list identity，例如 `Query.viewer`、`User:1.posts.ids`。

默认 records 是内存 `Map`。外部可以注入 `GraphDataRecordMap` 来接持久化、LRU、SIEVE 或自定义 TTL，但必须遵守 `GraphDataStore` 文档里的 records contract。

## TTL 当前语义

TTL 只控制 freshness，不清理内存。

- 写入时：`ttl > 0` 写成 `Date.now() + ttl`，`ttl <= 0` 写成 `0`。
- `expires === 0`：永不过期。
- `expires > Date.now()`：fresh。
- `expires <= Date.now()`：stale。
- record 不存在：missing。

stale 值仍可展示，`cache-and-network` 会后台校准。外部 eviction 删除 record 后，该地址从 stale/fresh 变成 missing，后续由 active selection refetch。

## 派生缓存

### Operation plan cache

源码：

- `packages/core/src/session/operation-cache.ts`
- `packages/core/src/sieve-cache.ts`

缓存内容：selection paths 到 GraphQL operation plan。

它适合 `SieveCache(128)`，因为：

- plan 可由 selection 重新计算。
- 丢弃不改变 UI 当前业务数据。
- key 可能随 selection shape 增长。

它不持久化，也不进入 `GraphDataRecordMap`。

### Completed freshness map

源码：

- `packages/core/src/session/query.ts`

用途：记录当前 query session 内已完成 operation 的 freshness，避免 TTL 内重复请求。

它是 session-local 状态：

- `refetch()` / `invalidate()` 会清空。
- 过期时 lazy delete。
- 不保存业务数据。
- 不跨页面或代码版本持久化。

### React / Solid session state

React session registry 只共享 `QuerySession`，不是业务数据缓存。

- 相同 scope / policy / ttl / metadata 才共享 session。
- release 后 registry 删除 session。
- 数据仍写入 provider 的 `GraphDataStore`。

Solid 当前更偏 primitive。多个 query / mutation 想共享业务数据，应传入同一个 `GraphDataStore`。

### Render / accessor memo

render tracking 和 accessor memo 只服务于当前 render 或一次访问链。

- 可随时重建。
- 不参与 persistence。
- 不进入 `GraphDataRecordMap`。

### EntityRef pool

EntityRef pool 用于稳定 `{ type, id }` 引用，减少无意义 signal update。

它不是业务数据，但会占内存。未来如果 `clear()` 表示用户边界切换，可以考虑同时释放 EntityRef pool。

## React / Solid 生命周期

React：

- provider 创建或接收一个 `GraphDataStore`。
- provider 不卸载时，同一 SPA 生命周期内可复用 store。
- 页面刷新、关闭 tab 后，默认内存 records 丢失。
- 传入持久化 records map 后，是否跨访问恢复由外部 records 实现决定。
- 适配器配置统一使用 `store`，不再保留长期 `cache` 双轨命名。

Solid：

- 不传 store 时，每次 primitive 调用可创建 call-local store。
- 需要共享 query / mutation 结果时，应由应用创建并传入同一个 store。
- 未来若提供 Solid provider，provider-local store 语义应与 React 对齐。

## 不建议

- 不要把业务数据 records 直接替换成无通知的 LRU / SIEVE。
- 不要把 TTL 过期当删除。
- 不要持久化 operation plan cache、completed freshness map、session registry、render/accessor memo。
- 不要让适配器默认持久化业务数据；持久化由外部 records map opt-in。

## 后续评估

- `GraphDataStore.clear()` 应确保旧 signals 被置空并通知 reader。
- `operationKey()` 若接受不稳定变量对象，应考虑 stable stringify。
- 可增加 dev-only stats：fields / slots size、EntityRef pool size、plan cache hit/miss、completed map size。
- 如出现真实大数据压力，再评估 family-aware eviction、分片恢复或按 scope 加载。
