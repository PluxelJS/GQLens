# 性能模型

本章描述 GQLens 的成本边界。核心目标不是“最少对象数”，而是让内存、订阅、更新通知的成本可估算。

## Accessor 不是数据对象

```ts
const user = q.user({ id });
```

`user` 返回的是 accessor node，不是 `User` 数据对象，也不是对象级 signal。

它只保存：

- 当前 GraphQL path：`Query.user(id)`
- schema contract field rule
- 可选 ref resolver：已知 `id` 时可解析为 `User:<id>`
- 子 accessor cache：避免同一 accessor 上重复创建相同 child accessor

它不会：

- 订阅整个 `User:<id>`
- 读取所有 User 字段
- 为 User 的所有 schema 字段创建 signal
- 在 User 任意字段变化时通知所有读过 `user` 的 reader

只有读取可读取终点时才进入响应式层：

```ts
q.user({ id }).name; // demand Query.user(id).name + read User:<id>.name signal
q.user({ id }).avatar; // demand Query.user(id).avatar + read User:<id>.avatar signal
q.user({ id }).posts({ first: 10 }).ids; // demand list identity + read ids signal
```

因此 `q.user({ id })` 本身只是路径镜头。字段 access 才是 signal subscription 边界。

## 成本公式

GQLens 的主要运行时成本应近似为：

```
cost =
  materialized entity field signals
  + materialized root / relation slot signals
  + active reader → signal subscriptions
  + active selection paths
  + accessor nodes reachable from active renders
```

不得退化为：

```
entity count × schema field count
```

schema 只决定 accessor shape；store signal 必须按读取或写入懒创建。

## Field Signal 粒度

实体字段是响应式通知单位：

```
User:1.name   → Signal<string>
User:1.avatar → Signal<string | null>
User:1.online → Signal<boolean>
```

字段级 signal 的收益：

- `User:1.name` 更新只通知读过 name 的 reader
- `User:1.avatar` 更新不影响只读 name 的 reader
- 不同 GraphQL 路径命中同一实体字段时共享同一个 signal

对象级 signal 的问题：

```
User:1 signal
```

一旦 `User:1.avatar` 变化，所有读过 `User:1.name`、`User:1.online`、`User:1.posts` 的 reader 都可能被唤醒。它节省 signal 数量，但扩大了重渲染范围。

GQLens 的选择是：**多一些懒创建的 field signal，换取更小的通知半径**。

## 懒创建不变量

- 不得为 schema 中所有字段 eager 创建 signal
- 不得为实体对象创建“全对象订阅”signal
- runtime 可以懒创建 field entry，但只有被读取或写入的 address 才 materialize
- `writeGraphQLResult()` 只写响应中实际出现的字段
- missing field 的 entry 可以存在，但不得被视为 cached

示例：

```
schema User: id, name, avatar, online, email, bio, posts
response User:1: id, name
```

cache 只需要：

```
User:1.id
User:1.name
```

若后续读取 `q.user({ id: "1" }).avatar`，可以创建 `User:1.avatar` entry，值为 `undefined`，并记录 active demand。不得因为读取 avatar 而创建 `online/email/bio/posts`。

## Reader 订阅成本

一个 reader 在一次 render 内可能多次读取同一个 signal。adapter 必须按 signal identity 去重：

```ts
q.user({ id }).name;
q.user({ id }).name;
```

订阅成本应为：

```
unique signals read by reader in latest committed render
```

而不是字段读取次数。

React adapter 还必须区分 draft render 与 committed render：

- render 阶段收集 draft signals / paths
- commit 后替换 reader 的 active selection
- 未 commit render 的 selection 不得污染 session
- cleanup 时取消旧 signal subscriptions

## Accessor Node 成本

生成 getter / 函数的 accessor node 是轻对象。它可以按 path 缓存在父 accessor 内，避免同一链路重复创建：

```
q.user({ id })          → User accessor
q.user({ id }).posts()  → PostList accessor
```

accessor cache 的 key 必须包含 field + canonical args。缓存 accessor node 不等于缓存数据值；数据仍来自 GraphDataStore signal。

需要避免的形态：

- 每次 scalar getter 都创建临时响应式对象
- 列表返回 entity object array
- child accessor 持有过期数据快照
- accessor node 自身成为订阅单位

## 列表性能边界

列表只订阅 identity：

```ts
const ids = q.todos({ done: false }).ids;
```

行字段通过显式 root accessor 读取：

```ts
q.todo({ id }).title;
q.todo({ id }).done;
```

这个设计避免列表组件订阅所有行字段。列表成员/顺序变化通知列表 reader；单行字段变化只通知对应字段 reader。

不得把列表设计成响应式 entity array：

```
Todo[] signal
```

否则任何行字段变化都可能污染列表 identity 的订阅边界。

## Args 与 Key 成本

field args 必须 canonicalize，保证：

```ts
q.user({ id: "1" }).name;
q.user({ id: "1" }).avatar;
```

命中同一 root + args path。

正确性上可以使用 canonical JSON；性能上可以优化：

- 对 args object 使用 WeakMap memo
- codegen / AST 插件预计算静态 path segment
- 对常见 scalar args 使用专门 key builder
- 对 ref by id 的 root field 快速解析 entity ref

这些只能优化 key 构建成本，不能改变 canonical equivalence。

## 与 GQty 的性能对照

GQty 通过 data proxy 与 normalized reactive cache 捕获访问，并强调 stable object references。它的成本集中在：

- proxy trap / access capture
- normalized object reference 维护
- object / tree subscription 语义
- scoped query selection batching
- array-like proxy 体验和 placeholder

GQLens 的成本集中在：

- field signal entry 数量
- reader → signal subscription 数量
- selection path / key 构建
- accessor node 数量
- slot / list identity signal 数量

两者都需要解决“读什么就订阅什么”的问题。GQLens 刻意把成本从 object/proxy reference 转移到 field signal + explicit identity slot，使性能边界更可度量。

## 性能不变量

- `q.user({ id })` 返回 accessor，不返回数据对象或对象级 signal
- scalar field / list identity 是响应式读取终点
- field signal 懒创建，按 entity field key 去重
- slot signal 懒创建，按 path + args 去重
- reader 对 signal 订阅按 signal identity 去重
- active selection 按 reader 最新 committed render 替换
- 列表 identity 与 entity field 不得合并成同一响应式单位
- key canonicalization 是正确性前提；预计算只是优化
