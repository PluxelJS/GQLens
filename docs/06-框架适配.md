# 06. 框架适配

核心对框架只暴露少量能力，适配层仅负责接入各框架的响应式模型。框架适配不应该重新定义 cache、selection 或 mutation 语义。

| 能力              | 说明                              |
| ----------------- | --------------------------------- |
| read field        | 执行字段 accessor，返回当前值     |
| track reader      | 将 reader 绑定到 field signal     |
| notify reader     | field signal 变更后通知该 reader  |
| cleanup selection | 组件卸载时清除其 active selection |

适配层的目标是让同一套 accessor API 在不同 UI runtime 下成立：

```ts
q.user({ id }).name;
q.todos({ done: false }).ids;
```

差异只能体现在“如何追踪 reader、如何触发重渲染、如何清理 selection”。

生成后 runtime 入口遵守宿主命名约束：React 使用 `useQuery()` / `useLiveQuery()`，Solid 使用 `createQuery()` / `createLiveQuery()`。不得设计裸 `query()` 作为公共入口；它语义过宽，且容易被理解为立即执行 transport。跨平台统一点是取得 `q` 之后的 accessor graph，以及 `defineSelection()` / `defineInvalidation()` 这类静态 selector builder。

适配层可以提供默认 HTTP fetch / WebSocket helper，但公共配置应优先接受外部 `fetcher` 和 `liveSubscriber`。协议选择、鉴权、重连、SSE / WS 偏好都属于应用边界；GQLens 的本职是把 operation、cache、selection 和 reader 语义接好。

## React

`useQuery()` 在 render 期间追踪组件内所有字段读取，并把这些 field signal 接入 React 的重渲染机制。任一被读字段变化时组件重渲染。不需要额外的 `useField` 包装。

React 适配的关键契约：

- 每次 render 得到一份新的 selection 快照
- commit 后这份快照成为当前 reader 的 active selection
- 未 commit 的 render 不应污染 session
- StrictMode 下重复 render 不应造成重复长期订阅
- 字段 signal 变化只通知读过该字段的组件

具体桥接可以使用 `useSyncExternalStore` 或等价订阅机制；设计约束是 commit / cleanup 语义正确，而不是绑定某个 Hook 名称。

这部分是 React adapter 最需要谨慎处理的边界。用户 API 可以保持简单，但设计上必须承认 concurrent rendering、条件分支和卸载清理的复杂性。

### 订阅粒度

```
列表组件   → 订阅 ID 列表
行组件     → 订阅单行字段
字段组件   → 订阅单字段
```

## Solid

Solid 的 fine-grained runtime 天然追踪 signal 访问，无需包装。适配层只需处理 query scope、selection batching、resource/suspense、cleanup。

Solid 的优势是字段 signal 可以直接进入响应式图，但 selection demand 仍然需要有明确的 scope。也就是说，Solid 可以少做订阅桥接，但不能跳过 QuerySession。

## API 对照

| 能力     | React                          | Solid                             |
| -------- | ------------------------------ | --------------------------------- |
| 查询     | `useQuery()`                   | `createQuery()`                   |
| 实时查询 | `useLiveQuery()`               | `createLiveQuery()`               |
| 字段读取 | `q.user({ id }).name`          | `q.user({ id }).name`             |
| Mutation | `useMutation(api.user.rename)` | `createMutation(api.user.rename)` |

表中的 API 是 codegen 暴露的生成后入口。底层 runtime hook 可以使用更明确的内部命名，例如 `useGQLensSession` / `createGQLensSession`，以避免和 codegen 生成的 accessor hook 互相递归或职责混淆。

Codegen 的框架差异由 adapter descriptor 表达：模块名、导入名、session hook 名和导出名是配置。accessor 生成逻辑不得写死某个框架包名。新增 Vue、Svelte 或其他运行时，应扩展 adapter，而不是复制一套生成器。
