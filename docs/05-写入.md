# 05. 写入

Mutation、live patch、optimistic update 均通过同一个 NormalizedCache 写入。

## Mutation

```ts
const rename = useMutation(api.user.rename)

await rename({
  id,
  name: "Alice",
  optimistic(cache) {
    cache.field({ type: "User", id }, "name").set("Alice")
  },
})
```

生命周期：

```
snapshot 旧值 → optimistic patch → 发送请求 → 成功? normalize+reconcile
                                                  失败? restore 旧值（首次写入则 invalidate）
```

成功后 optimistic 值自然被 server response 覆盖，无需额外处理。失败时自动回滚——实现上在 optimistic 回调执行前 snapshot 受影响字段的旧值，若旧值不存在则标记该字段为首次写入，失败后走 invalidate。

### 副作用：invalidates

mutation 的返回值有时无法覆盖所有受影响字段。例如 `addComment` 会改变 `post.commentCount`，但返回值可能不带这个字段。此时通过 `invalidates` 声明需要重新获取的范围。

codegen 根据 schema 生成类型安全的 `InvalidationSpec` union，确保类型名和字段名在编译期校验：

```ts
// 由 codegen 生成（根据 schema 中所有 entity 类型自动推导）：
type InvalidationSpec =
  | { type: "User", id: string, keys: ("name" | "avatar" | "online")[] }
  | { type: "Todo", id: string, keys: ("title" | "done")[] }
  | { type: "Post", id: string, keys: ("title" | "commentCount")[] }
  // ...

// mutation 中使用：
await addComment({
  postId,
  content: "...",
  invalidates: [{ type: "Post", id: postId, keys: ["commentCount"] }],
               // ^ "Post" 和 "commentCount" 均为字面量类型，拼写错误会在编译期报错
})
```

`invalidates` 将指定 entity+field 标记为 stale，下一次被读取时由 `cache-and-network` 策略自然触发 refetch——与 TTL 过期复用的是同一套「stale → read → refetch」流程，无需特殊路径。

## Optimistic Update

适合直接的字段变更：

```ts
cache.field({ type: "User", id }, "name").set(nextName)
cache.field({ type: "Todo", id }, "done").set(true)
```

不适合列表排序、成员增删等复杂边界变化。此时应 invalidate list identity。

## Invalidation

三种粒度：

```ts
cache.invalidate({ type: "User", id }, ["name"])    // 单个字段
cache.invalidate({ type: "Todo", id })               // 整个实体
q.invalidateRoot("todos", { done: false })           // root demand
```

| 变更类型           | 策略                    |
| ------------------ | ----------------------- |
| 字段值变化         | invalidate / patch 字段 |
| 实体整体变化       | invalidate 实体         |
| 列表成员/排序变化  | invalidate 列表 identity|
| 无法精确判断       | invalidate 相关 root demand |

## 冲突处理

1. optimistic 值先写入
2. 服务器返回后覆盖或确认
3. live patch 根据服务器 revision 判定新旧

推荐服务端返回 `revision` 字段：

```graphql
type User {
  id: ID!
  revision: Int!
  name: String!
}
```

无 `revision` 时，只能采用最后写入优先或按 operation 顺序保守处理。
