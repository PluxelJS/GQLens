# 05. 写入

Mutation、live patch、optimistic update 均通过同一个 NormalizedCache 写入。

写入侧的设计原则是：**能精确 patch 字段时就 patch 字段；不能精确判断影响范围时，就把相关 demand 标记为 stale**。不要让 mutation 结果直接绕过 cache 更新 UI。

## Mutation

生命周期：

```
snapshot 旧值 → optimistic patch → 发送请求 → 成功? normalize+reconcile
                                                  失败? restore 旧值（首次写入则 invalidate）
```

成功后 optimistic 值自然被 server response 覆盖，无需额外处理。失败时应回滚到旧值；若字段原本不存在，则标记为 stale，交给后续读取重新校准。

optimistic update 只适用于可逆、局部、由调用方明确指定的字段变更。列表排序、成员增删、权限变化、服务端派生字段应优先 invalidation。

### 副作用：invalidates

mutation 的返回值有时无法覆盖所有受影响字段。例如 `addComment` 会改变 `post.commentCount`，但返回值可能不带这个字段。此时通过 `invalidates` 声明需要重新获取的范围。

codegen 根据 schema 生成类型安全的 `InvalidationSpec` union：

```ts
type InvalidationSpec =
  | { type: "User"; id: string; keys: ("name" | "avatar" | "online")[] }
  | { type: "Todo"; id: string; keys: ("title" | "done")[] }
  | { type: "Post"; id: string; keys: ("title" | "commentCount")[] };
```

`invalidates` 将指定 entity+field 标记为 stale；若该字段仍属于 active demand，session 会按同一套 stale 调度模型 refetch。它与 TTL 过期复用的是同一套「stale → active demand → refetch」流程，无需特殊路径。

对于列表成员变化，invalidation 必须指向列表 identity，而不是随意猜测每个实体字段。底层稳定契约是 root / relation slot invalidation：

```ts
session.invalidateRoot("todos", { done: false });
```

生成层的目标 API 可以把它提升为 accessor selector：

```ts
invalidates: [(q) => q.todos({ done: false }).ids];
```

selector 只是一种类型安全表达方式；它必须被编译或解析为确定的 selection path / slot key，不能在 mutation 阶段重新引入任意运行时字段拦截。

## Optimistic Update

适合直接的字段变更：

```ts
cache.field({ type: "User", id }, "name").sig(nextName);
cache.field({ type: "Todo", id }, "done").sig(true);
```

不适合列表排序、成员增删等复杂边界变化。此时应 invalidate list identity。

## Invalidation

三种粒度：

```ts
cache.invalidate({ type: "User", id }, ["name"]); // 单个字段
cache.invalidate({ type: "Todo", id }); // 整个实体
session.invalidateRoot("todos", { done: false }); // root demand
```

| 变更类型          | 策略                        |
| ----------------- | --------------------------- |
| 字段值变化        | invalidate / patch 字段     |
| 实体整体变化      | invalidate 实体             |
| 列表成员/排序变化 | invalidate 列表 identity    |
| 无法精确判断      | invalidate 相关 root demand |

目标 API 可以让 invalidation 尽量靠近 accessor，而不是字符串：

```ts
q.invalidate((x) => x.todos({ done: false }).ids);
q.invalidate((x) => x.viewer);
q.invalidate((x) => x.post({ id }).commentCount);
```

无论表面 API 如何，落到 runtime 后都必须是 entity field、entity、root / relation slot 这三类确定影响范围。

## 冲突处理

1. optimistic 值先写入
2. 服务器返回后覆盖或确认
3. live patch 根据服务器 revision 判定新旧

推荐冲突判定契约：服务端返回单调递增的 `revision` 或等价版本字段。

```graphql
type User {
  id: ID!
  revision: Int!
  name: String!
}
```

无版本字段时，只能采用最后写入优先或按 operation 顺序保守处理。

冲突处理不得藏在 adapter 里。GQLens 可以提供默认策略，但服务端版本契约必须在设计上可见；实时协作、离线写入、多端编辑场景需要 revision / updatedAt 一类的顺序依据。
