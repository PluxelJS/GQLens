# 05. 写入

Mutation、live patch、optimistic update 均通过同一个 NormalizedCache 写入。

## Mutation

```ts
const rename = useMutation(api.user.rename)

await rename({
  id,
  name: "Alice",
  optimistic(cache) {
    cache.field({ type: "User", id }, "name").set("Alice")
  },
})
```

生命周期：

```
optimistic patch → 发送请求 → normalize 返回值 → reconcile → 必要时 invalidate
```

### 副作用：invalidates

mutation 的返回值有时无法覆盖所有受影响字段。例如 `addComment` 会改变 `post.commentCount`，但返回值可能不带这个字段。此时通过 `invalidates` 声明需要重新获取的范围：

```ts
await addComment({
  postId,
  content: "...",
  invalidates: [
    { type: "Post", id: postId, keys: ["commentCount"] },   // 精确字段
    { type: "Post", id: postId },                            // 整个实体
  ],
})
```

`invalidates` 触发当前 QuerySession 重新 selection-merge、发送请求、更新对应 field signal 的 `loading` 和 `data`。相比 Apollo 的 `refetchQueries`（需要按名称引用整条查询），`invalidates` 声明的是**数据需求**而非操作名，与 normalized cache 的 entity-field 模型一致，粒度更精确，不会过度请求。

## Optimistic Update

适合直接的字段变更：

```ts
cache.field({ type: "User", id }, "name").set(nextName)
cache.field({ type: "Todo", id }, "done").set(true)
```

不适合列表排序、成员增删等复杂边界变化。此时应 invalidate list identity。

## Invalidation

三种粒度：

```ts
cache.invalidate({ type: "User", id }, ["name"])    // 单个字段
cache.invalidate({ type: "Todo", id })               // 整个实体
session.invalidateRoot("todos", { done: false })     // root demand
```

| 变更类型           | 策略                    |
| ------------------ | ----------------------- |
| 字段值变化         | invalidate / patch 字段 |
| 实体整体变化       | invalidate 实体         |
| 列表成员/排序变化  | invalidate 列表 identity|
| 无法精确判断       | invalidate 相关 root demand |

## 冲突处理

1. optimistic 值先写入
2. 服务器返回后覆盖或确认
3. live patch 根据服务器 revision 判定新旧

推荐服务端返回 `revision` 字段：

```graphql
type User {
  id: ID!
  revision: Int!
  name: String!
}
```

无 `revision` 时，只能采用最后写入优先或按 operation 顺序保守处理。
