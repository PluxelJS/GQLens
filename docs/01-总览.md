# 01. 总览

## 定位

GQLens 是一个以 **reactive signal 为底层原语**的 GraphQL 客户端。它关心的不是“把一次 GraphQL 请求包装成 hook”，而是把 UI 中真实发生的字段读取，转化为可合并、可缓存、可精确更新的数据需求。

它解决四个问题：

1. 用类型安全的字段访问声明数据需求
2. 自动合并多个组件的 selection
3. 将返回数据写入 normalized cache
4. 字段变更后只通知依赖该字段的 reader

## 核心公式

```
field read = declare demand + read reactive value
```

对任何可读取终点（scalar 字段或列表 identity）的访问同时做两件事：

- **记录 selection demand**：向当前 QuerySession 声明「我依赖这条路径的数据」
- **读取 field signal**：从 NormalizedCache 订阅并返回对应实体字段的当前值

这两者不可拆分：UI 读到哪个可渲染值，selection 就声明到哪里；cache 中哪个字段变化，也只唤醒读过这个字段的 reader。关联字段本身只推进路径，直到下游字段或列表 identity 被读取时才形成具体 demand。

需要区分两类 reactive value：

- **Entity field**：例如 `User:1.name`，直接对应一个实体字段
- **Root / relation slot**：例如 `Query.viewer`、`User:1.posts(done:false)`，先保存“这条路径指向哪些实体”，再继续落到实体字段

这样 `q.viewer.name` 在首次读取时不必已经知道 viewer 是 `User:1`。它先声明 `Query.viewer.name`，等响应返回后，slot 解析到实体，再让后续读取共享 `User:1.name`。

## 为什么字段访问不是 Proxy trap

GQLens 受 GQty 启发：UI 读取字段，客户端收集 selection，并在 cache miss / stale 时自动请求。这个方向是 GQLens 的设计起点，而不是需要否定的东西。

`q.user({ id }).name` 与 Proxy 风格在调用形态上非常接近。GQLens 的差异不在表面语法，而在字段访问的运行时契约。

GQLens 选择基于 schema 生成 getter + 函数：

```ts
// 无参字段 → getter，无需括号
q.viewer.name;
q.viewer.avatar;

// 有参字段 → 函数，args 显式传入
q.user({ id }).name;
```

Proxy 方案也可以获得相似的链式体验：

```ts
query.user({ id }).name;
query.viewer.name;
```

但 Proxy 把字段模型建立在“任意属性读取”的运行时 trap 上；GQLens 把字段模型建立在 codegen 产物上。每个可访问字段都来自 schema metadata，只有生成出的 getter / 函数会产生 demand。

设计收益：

- schema 决定 accessor shape：字段是否存在、有无参数、返回 scalar / entity / list 均由生成代码表达
- 字段读取边界窄：业务字段读取不会与 `then`、`inspect`、展开、序列化、枚举等 JS 隐式属性访问混在同一个 trap 中
- 有参字段保持函数调用：args 是字段路径的一部分，必须显式进入 selection key
- 调试路径清晰：栈落在生成 getter / 函数，而不是统一的 Proxy trap
- AST / 编译优化稳定：插件可以直接识别 getter / call expression，不需要复原动态拦截语义

因此，这不是“getter 语法优于 Proxy 语法”的判断，而是：**字段访问应当是 schema-generated accessor contract，而不是 runtime universal trap**。

GQty 作为启发源和对照设计项，详见 [07-GQty 对照](./07-GQty对照.md)。本章只保留核心分歧。

## 系统分层

```
Accessor Graph      → codegen 生成的类型安全字段访问层
Runtime Collector   → 字段读取时收集 selection 路径
QuerySession        → 跨组件合并 active selection
Planner             → selection graph → GraphQL operation
Normalized Cache    → 实体字段级 signal 存储（含 TTL 驱逐）
Signal Runtime      → 字段级依赖追踪与精确通知
Transport           → Fetcher / LiveSubscriber 契约（协议由外部选择）
Compiler Plugin     → 可选：预计算 path、hash、静态诊断
```

## 传输模式

两种传输形态共享同一套 accessor API，区别仅在 scheduler：

- **Fetch**：一次性请求，受缓存策略（`cache-first` / `cache-and-network` / `network-only`）控制
- **Live**：消费持续订阅函数。GQLens 只关心 `subscribe(op, onData, onError) => unsubscribe`，WebSocket、SSE 或业务自定义通道都应由外部接入。

live 不是缓存策略。它只是持续推送数据写入同一套 NormalizedCache；协议、鉴权、重连和服务端 patch 形态不应被硬编码进核心。

默认缓存策略为 `cache-and-network`：先返回缓存，同时后台 fetch，兼顾即时性与时效性。

## Session 状态

生成后的 query accessor 同时暴露 session 状态：

```ts
const q = useQuery();

q.viewer.name; // 字段访问
q.loading; // boolean：存在任何 in-flight request 时为 true
q.error; // Error | null：最近一次请求错误
```

loading 和 error 是 session 级的——一次请求返回的所有字段共享同一组请求状态，不做字段级拆分。

`useQuery()` 指生成后 API。底层 framework adapter 使用更低层的 runtime hook；两层命名不得互相污染。

## 不变量

- 同一实体字段必须映射到同一个 signal
- root / relation slot 必须能稳定映射到实体引用或实体列表
- accessor node 不是数据对象，也不是对象级 signal
- 可读取终点必须同时产生 selection demand 和 signal subscription
- QuerySession scope 必须由 operation 语义决定，不得把不同 scope 的 selection 混合
- 组件卸载后必须移除其 active selection
- query / live / mutation 都必须汇入同一个 NormalizedCache
- AST 编译插件只能优化，不能成为正确性前提
